# 常用启动项工作台 UI 设计方案

## 1. 目标

这份设计主要解决两个问题：

- 现有 `常用启动项` 界面在条目增多后会失去可读性
- 如果要把它升级成真正的“工作台启动面板”，UI 应该怎么重构

## 2. 为什么现有 UI 不够用了

当前界面的核心问题不是功能少，而是结构上更像“左右两个列表拼在一起的工具窗”，它适合少量条目，但不适合扩容后继续承载：

- 左右等宽导致“扫描结果”和“常用启动项”视觉权重接近，但真实使用里右侧才是主工作区
- 启动项一多，平铺列表会迅速失控，无法承载分组、收藏、最近、失效提醒这些信息
- 条目只适合展示名称和路径，不适合扩展最近启动时间、标签、参数、状态、固定置顶等字段
- 界面偏“管理列表”，不够像“工作台启动面板”

## 3. 新 UI 的核心思路

建议把现有双栏结构升级为“三段式工作台布局”：

### 顶部工具条

承载全局操作：

- 搜索框
- 视图切换：全部、最近、收藏、失效项
- 快捷动作：手动添加、新建分组、批量管理
- 状态提示：当前分组、当前匹配数、索引状态

### 左侧导航栏

承担结构化导航，不再让主区自己扛所有层级：

- 全部启动项
- 最近启动
- 收藏
- 失效项
- 分组列表：开发工具、项目入口、运维脚本、目录快捷方式、临时工具

### 中间主工作区

这是核心区域，建议改成“卡片式工作台”。每个卡片展示：

- 名称
- 路径摘要
- 标签或备注
- 最近启动时间
- 状态标识：正常、失效、管理员、收藏
- 主动作：启动、打开目录、更多

### 右侧辅助区

右侧不再是另一个等价大列表，而是上下文面板：

- 当前选中启动项的详细信息
- 启动参数和备注
- 最近启动记录
- 快捷操作
- 搜索候选项加入当前分组的入口

## 4. 能力提升不只是“分组”

你指出的问题是对的。如果只是分组和一组启动，还是不够像工作台。UI 层真正的提升应该是：

- 把高频入口前置
- 把最近使用显式化
- 把收藏显式化
- 把失效项显式化
- 把“搜索结果加入工作台”做成顺手动作
- 让用户打开窗口后就像进入一个可直接开始工作的启动台，而不是进入一张维护表

## 5. 重点视觉能力

建议在第一版就明确以下视觉层级：

- 第一层：分组、卡片标题、主启动按钮
- 第二层：路径、标签、最近使用时间
- 第三层：启动参数、状态说明、历史记录

建议视觉风格是轻桌面工作台，而不是传统工具弹窗：

- 明亮中性色背景
- 清晰的面板分区
- 有层次的卡片
- 主动作高亮，但不过度花哨

## 6. 第一版最值得做的 UI 结构

如果只做一版最有感知的 UI，建议直接采用：

- 顶部全局搜索和视图切换
- 左侧导航栏
- 中间卡片式工作台
- 右侧详情与候选加入区
- 最近使用
- 收藏
- 分组导航
- 失效项标记
